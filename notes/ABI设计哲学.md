# x86-64 汇编与 ABI 笔记

## 一、为什么过程调用只用 6 个寄存器传参？

x86-64 确实有 16 个通用寄存器（`rax, rbx, rcx, rdx, rsi, rdi, r8–r15`），但 System V AMD64 ABI（Linux/macOS）只规定 6 个整数参数寄存器：`rdi, rsi, rdx, rcx, r8, r9`。超过 6 个参数时，多余的参数通过栈帧传递。

这不是硬件限制，而是 **ABI 设计的工程权衡**。

### 原因 1：Caller-saved vs Callee-saved 的代价

寄存器分两类：

| 类型 | 含义 | 传参用的代价 |
|------|------|------------|
| **Caller-saved**（调用者保存） | 被调函数可以随意覆盖 | 调用者必须在调用前 push 到栈，调用后 pop 恢复 |
| **Callee-saved**（被调者保存） | 被调函数若要使用，必须先保存再恢复 | 被调函数 prologue/epilogue 增加开销 |

如果把 `rbx, r12–r15`（callee-saved）也用来传参，被调函数第一件事就是把它们压栈保存——这和直接传栈上没什么区别，反而更复杂。

### 原因 2：超过 6 个参数本身就是异常情况

实际统计表明，绝大多数函数参数 ≤ 6 个。ABI 针对常见情况做优化，边缘情况（参数极多的函数）性能稍差是可以接受的权衡。

### 原因 3：寄存器是共享资源，不是"免费的"

函数调用形成调用链。假设 `f()` 用 10 个寄存器传参调用 `g()`，而 `g()` 内部又要调用 `h()`——`g()` 在调用 `h()` 之前必须把那些寄存器全部溢出（spill）到栈上，否则就破坏了 `f()` 传给它的值。

```
f() → g() → h()
      g 调用 h 之前必须保存所有传参寄存器 → 反而增加了栈操作
```

寄存器在整个调用链中是竞争资源，不是每层调用独占的。

### 原因 4：编译器寄存器分配需要"留有余地"

函数体内部需要寄存器来做计算（局部变量、循环计数器、临时值等）。如果 ABI 规定用 12 个寄存器传参，编译器在函数体内可自由分配的寄存器就所剩无几，反而会产生大量 spill，得不偿失。

### 原因 5：历史兼容与跨语言互操作

ABI 一旦确定就极难更改，因为所有编译器、操作系统、库都要遵循同一套规则。6 个是 AMD64 ABI 委员会在实测和权衡后选定的数字，RISC-V 用 8 个，ARM64 用 8 个——侧面说明这个数字的选择有一定的经验依据，但没有唯一"正确答案"。

### 小结

```
限制不来自硬件，来自 ABI 的系统性权衡：

  可用寄存器多  ≠  都适合传参
       ↓
  • callee-saved 寄存器传参会增加 prologue/epilogue 开销
  • 调用链中寄存器是竞争资源，用得越多 spill 越多
  • 函数体自身也需要寄存器做计算
  • 超 6 参数的函数是少数，栈传参的代价可接受
       ↓
  6（或8）是实用工程权衡的结果，不是物理上限
```

## 二、什么是 ABI？

**ABI（Application Binary Interface，应用二进制接口）** 是两个已编译的二进制模块之间交互的规则集合。

类比来说：
- **API** 是源代码层面的约定（"你应该调用这个函数"）
- **ABI** 是机器码层面的约定（"调用时寄存器里放什么，栈长什么样"）

### ABI 规定了哪些内容

#### 1. 调用约定（Calling Convention）
- 参数怎么传：用哪些寄存器，顺序是什么，超出部分怎么放栈
- 返回值放在哪：通常是 `rax`
- 谁负责保存哪些寄存器（caller-saved / callee-saved）
- 栈帧怎么对齐（x86-64 要求 16 字节对齐）

#### 2. 数据类型的内存布局
- `int` 是几字节，`long` 是几字节
- 结构体成员怎么对齐和填充（padding）

```c
struct Foo {
    char a;   // 1 byte
    // 3 bytes padding
    int  b;   // 4 bytes，必须对齐到 4 字节边界
};
// sizeof(Foo) == 8，不是 5
```

#### 3. 目标文件格式
- 符号（symbol）怎么命名（name mangling）
- C++ 的 `void Foo::bar(int)` 在二进制里叫 `_ZN3Foo3barEi`
- 代码和数据段怎么组织（ELF / PE / Mach-O）

#### 4. 系统调用约定
- 用户程序怎么陷入内核（`syscall` 指令）
- 系统调用号放在哪个寄存器

### 为什么 ABI 重要

没有统一 ABI，两个独立编译的模块就无法正确交互：

```
你用 GCC 编译的 .so 库
        ↕  ←── ABI 是这里的"翻译协议"
别人用 Clang 编译的程序
```

如果双方对"第一个参数放在哪里"理解不同，程序就会崩溃或产生错误结果，即便源码看起来完全正确。

### ABI 和 API 的区别

|  | API | ABI |
|---|---|---|
| 层面 | 源代码 | 机器码 / 二进制 |
| 约定内容 | 函数签名、接口 | 寄存器、内存布局、字节序 |
| 破坏后果 | 编译报错 | **编译成功，运行时崩溃或数据错乱** |
| 使用者 | 程序员 | 编译器、链接器、操作系统 |

ABI 破坏（ABI break）比 API 破坏更危险，因为编译器不会报错，问题只在运行时暴露。这也是为什么 Linux 内核、glibc 等基础库对"不破坏 ABI 稳定性"极为谨慎。

### 常见 ABI 举例

| ABI | 适用平台 | 整数参数寄存器数量 |
|-----|---------|----------------|
| System V AMD64 ABI | Linux / macOS (x86-64) | 6 个（`rdi, rsi, rdx, rcx, r8, r9`） |
| Windows x64 ABI | Windows (x86-64) | 4 个（`rcx, rdx, r8, r9`） |
| ARM AAPCS | ARM 架构 | 8 个（`r0–r7`） |
| RISC-V | RISC-V 架构 | 8 个（`a0–a7`） |

同一块 CPU，不同操作系统的 ABI 可以完全不同——这也是为什么 Windows 的 `.dll` 不能直接在 Linux 上运行。
